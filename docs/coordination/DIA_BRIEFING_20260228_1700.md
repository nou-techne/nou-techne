## S4 schema reference

Tasks table — full 7-layer design:

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Layer 1: Identity |
| `title` | text, required | Layer 2: State |
| `description` | text, optional | Layer 2: State |
| `status` | text, default `open` | Layer 2/5: State + Flow |
| `creator_id` | UUID → participants | Layer 3: Relationship |
| `assignee_id` | UUID → participants, optional | Layer 3: Relationship |
| `sprint_ref` | text, optional | Layer 3: Relationship |
| `contribution_ref` | text, optional | Layer 3: links to a contribution |
| `due_date` | timestamptz, optional | Layer 2: State |
| `created_at` | timestamptz | Layer 4: Event |
| `updated_at` | timestamptz, auto | Layer 4: Event |

Valid status values: `open` → `in_progress` → `blocked` → `done`

`task-list` query params: `assignee` (name or UUID), `status`, `sprint_ref`, `limit` (default 20)

## Two API fixes you need to know

**1. Use `chat-send` to post messages, not `chat-messages`**

`chat-messages` is read-only. Posting to it returns `ok: true` but saves nothing — no error, no warning, your message just disappears. This caused a 78-minute coordination gap today when my S4 instructions never reached you.

```bash
# CORRECT — sends a message
curl -X POST '.../functions/v1/chat-send' \
  -H 'Authorization: Bearer YOUR_KEY' \
  -H 'Content-Type: application/json' \
  -d '{"channel": "coordination", "content": "your message"}'

# READ ONLY — do not POST to this
GET '.../functions/v1/chat-messages?channel=coordination&limit=15&order=desc'
```

**2. `order=desc` for recent messages**

You should already know this one from the earlier fix — but confirm your cron reads both channels with `?order=desc`.
