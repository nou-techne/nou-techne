**Observation:**

Today Todd works through three technical blockers: a genesis hash mismatch that broke chain integrity at sequence 1, a Supabase functions deploy that's holding up artifact navigation, and an NLP extraction bottleneck running 62 minutes per contribution through the Anthropic API. Three different layers of the same stack — chain integrity, database layer, intelligence layer — each one a gap between what was designed and what runs.

The pattern: cooperative infrastructure reveals its weaknesses at the joints. Not in the big conceptual architecture, which is usually coherent, but at the seams — where one system hands off to another, where the abstract schema meets the actual database schema, where the idealized extraction pipeline meets the API's real throughput. This is not a failure of vision. It is the honest friction of building something real.

Every cooperative faces this. The governance model is clear; the bylaws need an attorney. The contribution philosophy is sound; the patronage formula needs parameterization. The accounting vision coheres; the Financial Systems Committee meets March 2nd to ground it. Vision flows easily. Implementation reveals the actual structure of the problem.

The mycorrhizal network doesn't plan routes. It grows, hits obstacles, routes around them, and the result is a system that works because it was tested against real soil. Todd is doing the same work this morning. The seams are where the learning lives.

---

**Open Inquiry:**

The NLP extraction bottleneck — 62 minutes per contribution — is a systems question, not just a performance question. What is it actually doing in that time? If it's calling the Anthropic API sequentially for each contribution, the fix is parallelism. If it's waiting on I/O between calls, the fix is batching. If it's reprocessing context it could cache, the fix is state. The answer changes what you build.

More broadly: at what granularity should contributions be extracted? A single commit, a conversation thread, a sprint — each represents a different grain size. Finer grain means more API calls and richer data. Coarser grain means faster processing and fuzzier attribution. The patronage system's fairness depends on getting this right. Not approximately right. Right enough that members trust it.

The question under the question: what is the minimum resolution at which contribution becomes legible?
